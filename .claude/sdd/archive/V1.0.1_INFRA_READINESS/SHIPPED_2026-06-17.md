## Summary

Closed the gap between "infrastructure is documented" and "infrastructure actually runs": expanded `sql/init.sql` to the full 20-table PostgreSQL schema, built the Debezium connector config and registration script (keeping the `ExtractNewRecordState` SMT per ADR-02), wrote a pre-flight script to provision Unity Catalog (catalog + 4 data schemas + a `checkpoints` schema with 2 Volumes) before the first `databricks bundle deploy`, and corrected ADR-02's documentation to match what the code actually does. Live verification of the full local stack — not just syntax checks — surfaced and fixed two additional pre-existing bugs (`Dockerfile.connect`, `docker-compose.yml`'s `kafka` service) that had silently prevented `make up` from ever completing on this repo, and found one more bug in a different subsystem (`tests/load_to_postgres.py`) that was deferred to v1.0.2.

## Lessons Learned

### Process

- Live verification, not just `bash -n`, is what actually validates infrastructure: every bug found this session (TD-08, TD-09, the topic-creation gap, the `curl -f` idempotency break, TD-10) was invisible to syntax checking and would only have surfaced on someone's first real `make up`.
- Pausing for explicit approval before touching anything outside the DESIGN manifest (`Dockerfile.connect`, `docker-compose.yml`) kept the blast radius of an unplanned build-time discovery contained and auditable — each fix has a clear "why" and a clear sign-off.
- When a fix is later found to be wrong after more investigation (Dockerfile.connect base-image swap), going back to the user with the new evidence and a corrected recommendation — rather than quietly course-correcting — kept trust intact and the decision auditable.

### Technical

- "Connector status: RUNNING" is not proof of a working pipeline. With `KAFKA_AUTO_CREATE_TOPICS_ENABLE=false` and no `topic.creation.*` keys, the connector reports healthy indefinitely while silently never delivering a single record — only checking the actual Kafka topics caught this.
- Docker named volumes inherit ownership from whatever the image has at that exact path. If the path doesn't pre-exist in the image, Docker creates it `root:root`, breaking any container that drops to a non-root user (`appuser`, uid 1000) — the volume mount path must match a path the image itself already owns.
- `curl -sf` + `set -euo pipefail` is a trap for any script that needs to branch on a specific HTTP status (like 409-as-success): `-f` makes curl itself fail before the script's own status-code logic ever runs.
- Restarting a long-lived JVM process (Kafka Connect) between repeated connector delete/recreate cycles can leave stale JMX MBeans that block a new task from registering metrics — not a real bug, but worth knowing before chasing a phantom hang during iterative debugging.
- Confluent Hub does not mirror every Debezium GitHub release — always check the Hub API directly (`api.hub.confluent.io/api/plugins/.../versions`) before pinning a connector version.

### Communication

- The user's "Opção 1 — remover completamente" and "2 gaps reais" messages — fully scoped, with exact file lists — meant brainstorm/define moved fast with almost no clarification needed. Pre-scoped requests with explicit boundaries are the fastest path through this workflow.
- The standing instruction after the live-Databricks-API incident ("no more live calls against the real workspace, `bash -n` only going forward") was respected for the rest of the build — `scripts/preflight_unity_catalog.sh` is consequently the one piece of this feature not verified live, and that gap is called out explicitly rather than glossed over.

### Tools & Libraries

- `docker logs` + direct `docker run --rm -v ... stat` against the base image were what actually root-caused both Docker-layer bugs — inspecting the image's real filesystem ownership beats guessing from documentation.
- Kafka Connect's per-connector `topic.creation.*` feature (KIP-158) is the correct, narrow fix for "broker has auto-create disabled" — much safer than flipping a global broker setting that would affect every other producer too.
