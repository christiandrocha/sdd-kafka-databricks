## Summary

Unity Catalog Lineage atribui linhagem por caminho de notebook, não por execução
parametrizada — as 20 execuções de `pipeline_bronze.ipynb` e as 11 de
`pipeline_silver.ipynb` apareciam agrupadas sob um único nó cada, em vez de 31 nós
distintos. Uma investigação anterior já tinha descartado bug de código como causa; o
problema era estrutural ao próprio modelo de execução por notebook parametrizado.

Esta feature migra Bronze (20 domínios) e 10 dos 11 domínios Silver — `users_mongo` e
`users_mssql` continuam só com Bronze, já que seu Silver é o FULL OUTER JOIN bespoke de
`pipeline_users.ipynb` — para um único pipeline Databricks Lakeflow Declarative Pipelines
(`pipelines/bronze_silver_dlt.py`), gerado em loop sobre `contracts/*.yml`, substituindo 30
tasks DABs por 1 `pipeline_task`, em `dev`/`prod`. `free_edition` (sem suporte verificado a
Lakeflow serverless), os 6 notebooks Gold e `pipeline_users.ipynb` ficam deliberadamente
fora do escopo — nenhum deles tem o problema de linhagem que esta migração resolve. A
decisão sobrescreve, de forma documentada, a rejeição original do ADR-03 a DLT/Lakeflow,
mas só para a parte do pipeline onde o trade-off agora compensa.

O ciclo completo (brainstorm → define → design → build) incluiu pesquisa ao vivo via
`WebSearch`/`WebFetch` contra a documentação atual do Databricks antes de fixar o file
manifest — confirmando, entre outras coisas, que `apply_changes()` foi renomeado para
`create_auto_cdc_flow()` e que o módulo `dlt` foi renomeado para `pyspark.pipelines`.

## Lessons Learned

### Process

- Pesquisar as premissas técnicas (A-001 a A-005) via `WebSearch`/`WebFetch` *antes* de
  fixar o file manifest no DESIGN evitou descobrir, só no meio do build, que `apply_changes`
  tinha sido renomeado ou que o módulo `dlt` virou `pyspark.pipelines` — esses achados
  mudaram decisões de design (Decision 1 e 2), não só detalhes de implementação. Pesquisar
  antes do DESIGN, não durante o BUILD, foi o ponto certo desta vez.
- Mesmo com uma DESIGN detalhada, reler o código-fonte real (`contracts/*.yml`, a estrutura
  de `databricks.yml`) durante o BUILD revelou uma exclusão necessária (`users_mongo`/
  `users_mssql`) que o esboço do DESIGN não tinha deixado explícita. Confirma o padrão já
  visto em `GOLD_DIMENSION_JOIN_INTEGRITY`: o DESIGN é um guia, não um substituto para ler o
  código de verdade no momento do build.

### Technical

- `ruff check <arquivo explícito>` ignora o `exclude` do `pyproject.toml`; só
  `ruff check .` (sem caminho explícito) o respeita. Útil saber isso antes de concluir
  erroneamente que um arquivo precisa de `# noqa` em vez de só ajustar o exclude.
  Curiosamente, `pipelines/` precisava exatamente da mesma exceção que `notebooks/` já
  tinha — `spark`/`dbutils` como globals injetados pelo runtime Databricks, não nomes
  realmente indefinidos.
  > Tag: **[[note-ruff-exclude-ignored-with-explicit-path]]**
- A simetria que `databricks.yml` mantinha entre `dev`/`prod`/`free_edition` (mesmas
  `task_definitions`, computação diferente) quebrou pela primeira vez nesta feature —
  `dev`/`prod` agora rodam código genuinamente diferente de `free_edition` para
  Bronze/Silver, não só configuração de cluster diferente. Documentado explicitamente no
  novo ADR como uma consequência negativa aceita, não um efeito colateral não examinado.

### Communication

- O usuário pediu explicitamente, no brainstorm, para considerar 4 perguntas específicas
  (mapeamento de contrato para DLT, estratégia de parametrização, mudanças no
  `databricks.yml`, se Gold deveria migrar) — ter essas perguntas registradas literalmente
  no BRAINSTORM tornou trivial confirmar, no DESIGN, que todas as 4 tinham sido respondidas
  antes de prosseguir para o build.

### Tools & Libraries

- `WebSearch`/`WebFetch` contra `docs.databricks.com` foi essencial aqui porque o
  conhecimento de treinamento sobre "DLT" estava desatualizado em relação ao rebrand para
  "Lakeflow" e à troca de `dlt` por `pyspark.pipelines` — sem essa pesquisa ao vivo, o
  DESIGN teria sido escrito inteiramente contra uma API já parcialmente obsoleta.

## Recommendations for Future Work

| Area | Recommendation |
|------|----------------|
| Validação live | Antes de considerar esta feature "pronta para produção" (e não só "pronta para revisão"), rodar `databricks bundle deploy -t dev` + `databricks bundle run` de fato contra um workspace real, e confirmar AT-001/002/003/004/006 do DEFINE, que ficaram marcados como não verificáveis neste sandbox |
| Numeração de ADRs | Mesma dívida já flagada em `GOLD_DIMENSION_JOIN_INTEGRITY`: `docs/adr/006_lakeflow_migration.md` é sequencial só dentro da própria pasta, não corresponde a nenhum "ADR-06" informal já citado em `CLAUDE.md`. Vale uma sessão de `/design` dedicada a renumerar tudo de uma vez |
| `free_edition` + Lakeflow | O ADR desta feature deixa explícito que `free_edition` não migrou por falta de confirmação sobre suporte a pipelines serverless no Free Edition — vale investigar isso como uma feature futura, não assumir que nunca será possível |
