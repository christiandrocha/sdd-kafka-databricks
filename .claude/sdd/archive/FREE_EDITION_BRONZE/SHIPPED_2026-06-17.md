## Summary

Databricks Free Edition restringe o compute serverless a uma allowlist de rede fixa, não
customizável — o que tornava o `pipeline_bronze.ipynb` original (Kafka streaming via
`spark.readStream`) inexecutável lá. Esta feature adicionou um segundo modo de ingestão
(`source_mode=volume`): um script Python exporta os 20 tópicos Kafka pós-SMT para Parquet,
um Unity Catalog Volume novo (`landing.kafka_export`) recebe esses arquivos, e o Bronze passa
a ler em batch quando `source_mode=volume`, reaproveitando 100% da lógica de contrato/DDL/MERGE
já existente. No caminho, a feature também resolveu uma incompatibilidade de compute mais
ampla — Free Edition só aceita serverless, mas todos os 37 tasks do `databricks.yml`
referenciavam um cluster clássico — reestruturando o bundle com YAML anchors para que `dev`,
`prod` e o novo target `free_edition` coexistam sem duplicar conteúdo.

## Lessons Learned

### Process

- Catching the classic-vs-serverless compute incompatibility during `/design` — before any
  code was written — instead of during `/build` saved a full restructuring cycle. The same
  "always verify the platform's real constraints before committing to an architecture"
  habit that paid off in v1.0.1 (Dockerfile.connect, kafka volume) paid off again here, just
  one phase earlier.
- Live-testing a brand-new script against the real local stack — rather than stopping at
  `ruff check`/`bash -n` — surfaced 4 real, otherwise-invisible bugs in this build alone (the
  Kafka listener, 6 contract type mismatches, the timestamp wire-format assumption, and the
  unfiltered `__deleted` field). None of these would have shown up in a syntax-only review.
- Pausing for one consolidated approval covering all 6 contract type fixes (instead of
  asking once per field) kept the interruption count proportional to the decision, not to
  the number of lines touched — the user engaged with it as a single, well-evidenced batch.

### Technical

- Databricks Free Edition's outbound network restriction (a fixed, non-customizable
  allowlist) is the real, defining constraint for this whole feature — it's not relaxable
  by anything DABs/CLI-side, only by changing what the workspace needs to reach.
- DABs has no mechanism to exclude a root-level resource from a single target
  (`databricks/cli#2872`, open). The correct workaround — define each job entirely inside
  its own target's `resources:` block, sharing task bodies via YAML anchors — turned out to
  be a documented, working community pattern, not an improvisation.
- The checkpoint in Structured Streaming was never the source of Bronze's idempotency — the
  `MERGE INTO ... WHEN NOT MATCHED` clause was always doing that work. Realizing this meant
  the batch (`volume`) mode needed zero new idempotency logic.
- Debezium's wire format for timestamps depends on the source column's SQL type:
  `TIMESTAMPTZ` → ISO-8601 string (`ZonedTimestamp`), plain `TIMESTAMP` under
  `time.precision.mode=connect` → epoch-millis long. Code that assumes only one of these
  will break silently on the other — worth remembering for any future Debezium consumer.
- A contract's declared type is only as trustworthy as the first piece of code that actually
  enforces it. `to_create_table_ddl()` had been generating DDL from these 6 wrong types since
  v1.0.0 without anyone noticing, because nothing had cast real decoded values against them
  until this feature's export script did.

### Communication

- The user's direct questions ("Qual seria o mais recomendado?") after rejecting an
  open-ended `AskUserQuestion` were a clear, consistent signal: lead with a recommendation
  and a one-paragraph rationale first, and only fall back to multiple-choice when the
  decision genuinely has comparable trade-offs worth presenting side by side.
- Bringing the DABs compute-exclusion limitation back to the user mid-design, with the
  GitHub issue number as evidence, kept the "approved direction, corrected mechanics" framing
  honest — it wasn't a new ask, just the real way to deliver what was already agreed.

### Tools & Libraries

- `confluent-kafka`'s `DeserializingConsumer` + `AvroDeserializer(schema_str=None)` reuses
  whatever schema ID is embedded in each message's wire format — no need to manually fetch
  the Avro schema string the way the notebook's Kafka-streaming path does via raw `requests`.
- `pyarrow.Table.from_pylist(records, schema=...)` is strict about per-field types — it does
  not silently coerce a string into an int32 column, which is exactly the assertion that
  caught all 6 contract bugs and the timestamp wire-format bug in this build.
