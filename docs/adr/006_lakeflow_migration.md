# ADR 006 — Bronze+Silver Migrate to Lakeflow Declarative Pipelines (Overrides ADR-03 Scope)

**Status:** Accepted
**Date:** 2026-06-18

## Context

Unity Catalog Lineage attributes lineage by *notebook path*, not by *parametrized run*. The
20 executions of `pipeline_bronze.ipynb` (one per domain) and the 11 executions of
`pipeline_silver.ipynb` therefore appear grouped under a single notebook node each, instead
of as 31 distinct table-level lineage edges. A prior code audit
(`GOLD_DIMENSION_JOIN_INTEGRITY`-adjacent "Bronze reading Bronze" investigation) found no bug
in the notebooks or `databricks.yml` — the cause is structural to the parametrized-notebook
execution model itself, not fixable without changing how Bronze/Silver execute.

This directly reopens a decision ADR-03 already made: `.claude/03_design.md` records
`alternatives_considered: [..., "Lakeflow DLT pipelines (rejected — different abstraction,
less explicit control)"]`. That rejection was correct for the trade-offs known at the time —
this ADR documents why the trade-off now reads differently, and scopes exactly how far the
reversal goes.

## Decision

Migrate Bronze (20 domains) and 10 of the 11 Silver domains to a single Lakeflow Spark
Declarative Pipeline (`pipelines/bronze_silver_dlt.py`), generated by looping over
`contracts/*.yml`, for the `dev` and `prod` targets only.

**Explicitly NOT migrated** (ADR-03's original rejection still stands for these):
- `free_edition` — keeps running `pipeline_bronze.ipynb`/`pipeline_silver.ipynb`
  (`source_mode=volume`) until Lakeflow serverless pipeline support there is verified.
- The 6 Gold notebooks — each is already 1 notebook : 1 execution, so the lineage-grouping
  problem doesn't apply to them; their imperative logic (two-stage aggregation in
  `gold_revenue_per_restaurant`, `row_number()` dedup guards) is exactly the kind of explicit
  control ADR-03 said DLT lacks.
- `pipeline_users.ipynb` (`silver_users` task) — same reasoning as Gold: one task, no lineage
  bug, and its `FULL OUTER JOIN` + full-refresh (`mode("overwrite")`) doesn't fit Lakeflow's
  incremental streaming-table model.

## Rationale

**Why revisit ADR-03 now:** the lineage cost is real and confirmed (not hypothetical) — a
prior code audit explicitly ruled out a code bug as the cause, leaving "the execution model
itself" as the only remaining explanation. ADR-03's "less explicit control" trade-off was
weighed against a hypothetical cost at the time; it's now weighed against a demonstrated one.

**Why not migrate everything:** the lineage-grouping problem specifically affects notebooks
that are *parametrized and executed many times*. Gold and `silver_users` are each already
executed exactly once per task — migrating them would take on DLT's "less explicit control"
trade-off for a problem they don't have.

**Key API findings that shaped the implementation** (verified via Databricks documentation,
2026-06-18, not asserted from memory):
- The Python module was renamed: `dlt` → `pyspark.pipelines` (`@dlt.table` → `@dp.table`,
  etc.). The legacy `dlt` alias still works; this migration uses the current API.
- `apply_changes()` is deprecated in favor of `create_auto_cdc_flow()` (same signature) —
  used for the `merge_key` upsert logic in both Bronze and Silver, replacing the hand-written
  `MERGE INTO` from the notebooks. For the current test dataset (no real `op='u'` events),
  this produces identical output to today's pipeline; in general it is *more* correct, since
  Bronze's current `WHEN NOT MATCHED THEN INSERT`-only logic would silently drop a genuine
  row update where `create_auto_cdc_flow` would apply it.
- `check: unique` (the cross-row uniqueness rule from `GOLD_DIMENSION_JOIN_INTEGRITY`) has no
  native `@dp.expect` equivalent — it's implemented as a stream-static join inside the
  quarantine table's function body, reading `spark.read.table(silver_table)` fresh per
  trigger, the same property `process_silver_batch()`'s `spark.table(silver_table)` call
  relied on in the notebook version.
- Quarantine routing has no native DLT action either (`@dp.expect*` only warns, drops, or
  fails the pipeline) — implemented as a second `@dp.table` per domain, reading the same
  upstream candidate view with the inverse predicate, so nothing is silently dropped.

## Alternatives Considered

1. **Leave Bronze/Silver as parametrized notebooks, accept the lineage limitation
   permanently** — rejected: the lineage graph becoming illegible for 31 domains is a real,
   ongoing cost to anyone auditing the pipeline.
2. **Migrate Gold and `silver_users` too, for architectural consistency** — rejected: neither
   has the lineage problem this migration solves, and both would lose real, recently-shipped
   imperative logic for no corresponding benefit.
3. **Migrate `free_edition` in the same pass** — rejected: Lakeflow serverless pipeline
   support there is unverified; deferred to a follow-up once confirmed, rather than blocking
   this migration on it.

## Consequences

**Positive:** each of the 31 migrated domains is now its own lineage node in Unity Catalog.
`dlt.apply_changes`/`create_auto_cdc_flow` removes 31 hand-written `MERGE INTO` statements'
worth of duplicated upsert logic. Lakeflow's native pipeline event log gives quality-rule
pass/fail metrics that didn't exist before.

**Negative:** `dev`/`prod` and `free_edition` now run genuinely different code for
Bronze/Silver, not just different compute settings on the same code — this breaks the
symmetry ADR-06 (`databricks.yml` task anchors shared across all 3 targets) relied on for
that part of the pipeline specifically. The stream-static join implementation of
`check: unique` and the dynamic per-domain table registration loop were not verified against
a live Databricks workspace in this session — see
`.claude/sdd/reports/BUILD_REPORT_LAKEFLOW_MIGRATION.md` for what was and wasn't validated.
